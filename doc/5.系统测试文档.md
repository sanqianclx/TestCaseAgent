## 1. 测试概述

### 1.1 测试目标

TestGenerate Agent 是一个基于 LLM 的智能测试生成系统，其测试目标涵盖两个层面：

1. **系统自身的功能测试**：验证系统各模块（工作流、工具、语言适配器、Web 服务）按设计正确运行。
2. **系统生成产物的质量测试**：验证系统生成的测试代码可执行、能正确暴露源代码缺陷、覆盖率达标。

本文档记录系统在开发过程中采用的测试策略、测试用例、测试结果与缺陷修复情况。

### 1.2 测试范围

| 测试维度 | 范围 |
| :--- | :--- |
| 功能测试 | 三语言测试生成全流程、自愈循环、覆盖率测量、产物导出 |
| 接口测试 | Web 后端 REST API、SSE 流式推送 |
| 集成测试 | CLI 三种模式、Web 端 Agent/Workflow 双模式、数据库持久化 |
| 兼容性测试 | Windows / 跨工具链（MSYS2 g++、Maven、pytest） |
| 质量验证 | 生成测试代码的可执行性、缺陷暴露能力、覆盖率准确性 |

### 1.3 测试环境

| 项目 | 配置 |
| :--- | :--- |
| 操作系统 | Windows 11 Home China 10.0.26100 |
| Node.js | >= 22 |
| Python | >= 3.10，pytest >= 8.0.0 |
| Java | 17+，Maven（JUnit 5） |
| C++ | MSYS2 UCRT64 g++ 16.1.0，支持 C++17 |
| 数据库 | MySQL 8.0+ |
| LLM | DeepSeek（deepseek-v4-flash / deepseek-v4-pro） |
| 浏览器 | Chrome / Edge |

> **说明**：系统未引入自动化单元测试框架（package.json 无 `test` 脚本），测试以**功能验证 + 集成测试 + 真实产物验证**为主，通过实际运行 CLI 与 Web 界面、检查输出产物来验证功能正确性。本测试文档如实记录这一情况。

## 2. 测试策略

### 2.1 测试分层

```
┌─────────────────────────────────────────────┐
│  L4 产物质量验证（生成测试能否抓 bug/覆盖率） │
├─────────────────────────────────────────────┤
│  L3 端到端集成测试（CLI 三模式 / Web 双模式） │
├─────────────────────────────────────────────┤
│  L2 接口测试（REST API / SSE / 数据库）       │
├─────────────────────────────────────────────┤
│  L1 模块功能测试（工作流 / 工具 / 适配器）     │
└─────────────────────────────────────────────┘
```

### 2.2 测试方法

| 方法 | 说明 |
| :--- | :--- |
| 黑盒功能测试 | 通过 CLI/Web 界面输入源代码，检查输出产物是否符合预期 |
| 接口测试 | 用 HTTP 请求验证各 REST 端点的响应格式与状态码 |
| 产物验证 | 检查生成的测试代码可独立编译执行、报告内容完整、覆盖率数据准确 |
| 故障注入 | 输入含已知缺陷的源代码，验证生成的测试能否暴露缺陷 |
| 异常路径测试 | 缺失依赖、超时、无测试符号、LLM Key 失效等边界场景 |

## 6. 产物质量验证

### 6.1 验证方法

系统生成的测试产物存放在 `output/exports/` 目录。本节通过实际产物验证系统生成测试的**可执行性**与**缺陷暴露能力**。

### 6.2 C++ 缺陷暴露验证

**被测文件**：`test_source_small.cpp`（含 `findMax` 正确实现 + `removeOddNumbers` 含 erase-in-loop 索引跳跃缺陷）

**生成产物**：`output/exports/agent/cpp/` 下的测试代码与报告

**验证结果**：

| 指标 | 数值 |
| :--- | :--- |
| 生成测试用例数 | 12 个 |
| 通过 | 10 个 |
| 失败 | 2 个 |
| 被测代码行覆盖率 | 100.00%（13/13 行） |

**缺陷暴露情况**：生成的测试成功暴露了 `removeOddNumbers` 的索引跳跃缺陷——全奇数场景 `{1,3,5,7}` 期望空向量，实际返回 `{3,7}`（漏删）；相邻奇数场景 `{1,3,6,7,9,10}` 期望 `{6,10}`，实际返回 `{3,6,9,10}`。报告自动给出根因分析与修复方案（erase-remove idiom）。

> **结论**：系统生成的测试不仅可执行，而且具备真实的缺陷检测能力，覆盖率达标，符合反迎合测试设计目标。

### 6.3 Java 产物验证

**生成产物**：`output/exports/agent/java3/` 含完整 Maven 工程结构（`src/main/java`、`src/test/java`、`pom.xml`、`target/` 编译产物）

**验证结果**：测试代码可独立 `mvn verify` 执行，JaCoCo 覆盖率报告生成于 `target/site/jacoco/`，测试计划与测试代码分别导出。

## 7. 缺陷与修复记录

开发过程中发现并修复的代表性问题：

| 编号 | 问题描述 | 根因 | 修复方案 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| BG-01 | LLM 诊断倾向于无脑返回"重生成测试代码" | 单一决策 LLM 易误判 | 引入 `reconcileNextAction()` 规则校正，按 diagnosis_type 重映射 | ✅ 已修复 |
| BG-02 | LLM 输出 JSON 被截断导致解析失败 | maxOutputTokens 不足 | 实现 `continueTestCaseJson()` 续写 + 重叠拼接 + LLM 修复 | ✅ 已修复 |
| BG-03 | JaCoCo 取到 sourcefile 局部覆盖率值 | xml 解析层级错误 | `extractXmlCounter()` 取最大 counter，缺 xml 时抢救跑 report | ✅ 已修复 |
| BG-04 | C++ 在 Windows 找不到 g++/gtest | PATH 未含 MSYS2 工具链 | `resolveToolchainBinDirs()` 解析 UCRT64 bin 并前置 PATH | ✅ 已修复 |
| BG-05 | CLI 彩色输出在管道/CI 产生 ANSI 乱码 | 直接用 ANSI 码 | 重构为显式 API，检测 NO_COLOR/CI/非 TTY 降级 | ✅ 已修复 |
| BG-06 | BigInt 序列化导致 API 响应异常 | JSON.stringify 不支持 BigInt | `safeJsonStringify()` 自定义序列化 | ✅ 已修复 |
| BG-07 | Java surefire 跨行 failure 块解析失败 | 正则未处理跨行 | 两条正则分别处理跨行块与自闭合用例 | ✅ 已修复 |
| BG-08 | 任务取消后工作流仍继续执行 | 无取消检查点 | `throwIfTaskRunCancelled()` 在关键节点检查标志 | ✅ 已修复 |

| 项目 | 说明 |
