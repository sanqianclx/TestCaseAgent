## 1. 系统概述

### 1.1 项目定位

TestGenerate Agent 是一个基于 **Mastra Agent 框架 + DeepSeek LLM** 的多语言智能测试生成系统。系统以源代码文件为输入，经自主决策与工具调用，全自动产出可执行的测试代码与结构化测试报告，支持 **Python（pytest）、Java（JUnit 5）、C++（GoogleTest）** 三种语言一体化生成。

系统的核心差异化设计是**反迎合测试哲学**：测试用例的使命是"暴露源代码缺陷"而非"证明源代码正确"。Agent 在设计用例与生成测试代码时，被明确禁止迎合源代码中的 bug（如用 `assertThrows` 接住本应抛出的异常、篡改输入绕过缺陷、修改期望值匹配错误输出），从而保证生成的测试真正具备缺陷检测能力。

### 1.2 核心能力

| 能力域 | 说明 |
| :--- | :--- |
| 自主规划 | Agent 根据输入源代码自行判断工作路径，支持自然语言交互与自主 REPL 模式 |
| 多语言支持 | Python / Java / C++ 一行切换，按文件扩展名自动检测 |
| 工具调用 | LLM 负责决策，工具负责执行（AST 解析、测试执行、覆盖率测量、产物导出） |
| 自愈循环 | 测试失败时自动诊断根因 → 修正测试代码 → 重新执行，最多可配置 20 次 |
| 真实覆盖率 | Python coverage.py / Java JaCoCo / C++ gcov，失败时降级到符号覆盖率 |
| 一键导出 | 生成可执行测试代码 + Markdown 测试报告（含覆盖率、版本历史、诊断详情） |
| Web 管理 | Express + Prisma + MySQL 后端，React + Ant Design 前端，支持会话/任务/文件/工作空间管理 |

### 1.3 技术栈

| 层次 | 技术 | 说明 |
| :--- | :--- | :--- |
| Agent 框架 | Mastra（TypeScript） | Agent 编排、Workflow 调度、工具注册、Studio 可视化调试 |
| LLM | DeepSeek（`@ai-sdk/deepseek`） | 快慢双通道：`deepseek-v4-flash`（首次快速生成）+ `deepseek-v4-pro`（自愈重试高精度） |
| 代码解析 | Python `ast` + Tree-sitter | Python 走标准库 `ast`；Java/C++ 走 `tree-sitter-java` / `tree-sitter-cpp` |
| 测试执行 | pytest / Maven（JUnit 5）/ g++（GoogleTest） | 在临时工作目录中以子进程方式隔离执行，自动超时保护 |
| 覆盖率 | coverage.py / JaCoCo / gcov | 真实行/分支覆盖率，失败降级到符号覆盖率 |
| Web 后端 | Express + Prisma + MySQL | REST API、JWT 鉴权，管理会话/任务/文件/工作空间 |
| Web 前端 | React + Vite + Ant Design | 仪表盘、对话界面、文件与工作空间管理 |
| Schema 校验 | Zod | 工具入参出参严格校验 |
| 内存管理 | 内置 InMemoryStore | 会话记忆、上下文摘要、事实存储 |

## 2. 系统总体架构

### 2.1 架构分层

系统采用**分层 + 多 Agent 协作**架构，自下而上分为五层：

```
┌─────────────────────────────────────────────────────────────┐
│                      交互接入层                              │
│   CLI（命令行/交互/自主）  │  Web 前端（React+AntD）         │
└──────────────┬──────────────┬────────────────────────────────┘
               │              │ SSE / REST
┌──────────────▼──────────────▼────────────────────────────────┐
│                   Web 服务层（Express）                       │
│  鉴权中间件 │ 业务 Controller │ 统一响应 │ 流式推送(SSE)       │
└──────────────┬───────────────────────────────────────────────┘
               │
┌──────────────▼───────────────────────────────────────────────┐
│              编排层（Mastra Agents + Workflow）               │
│  7 个 Agent │ 7 步工作流 │ 自愈循环 │ 诊断决策                  │
└──────────────┬───────────────────────────────────────────────┘
               │
┌──────────────▼───────────────────────────────────────────────┐
│            工具与语言适配层（Tools + Adapters）                │
│  parse/execute/coverage/export 等 7 工具 │ 3 语言适配器         │
└──────────────┬───────────────────────────────────────────────┘
               │
┌──────────────▼───────────────────────────────────────────────┐
│                运行时与持久化层                                │
│  Python 桥接 │ 命令执行器 │ MySQL(Prisma) │ 日志 │ 内存存储     │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 核心工作流

系统的核心是 `src/mastra/workflows/generate-test-workflow.ts` 中定义的 7 步串行工作流：

```
输入源文件 → ①读取并解析 → ②设计测试用例 → ③导出测试计划
          → ④生成测试代码 → ⑤执行测试 → ⑥自愈循环 → ⑦导出最终产物
```

各步骤说明：

| 步骤 | 名称 | 实现要点 |
| :--- | :--- | :--- |
| ① | read-parse-source | 读取源文件 → `detectLanguage()` 检测语言 → 取语言适配器 → `adapter.parseSource()` 得到结构化符号表 |
| ② | design-test-cases | 逐符号分批调用 `testCaseAgent` 设计测试用例，支持中英文数字用例数限制解析、JSON 截断续写与修复 |
| ③ | export-cases-plan | 导出"计划版"用例 Markdown，不写测试代码，便于人工预审 |
| ④ | generate-test-code | 调用 `testCodeAgent` 生成第一版测试代码，`extractCode()` 剥离代码围栏 |
| ⑤ | execute-tests | 适配器在临时工作目录中以子进程方式真实执行测试，返回 `ExecutionResult`（通过/失败/错误数、stdout/stderr） |
| ⑥ | self-healing | 若未通过，进入自愈循环：诊断根因 → 决策是否重生成 → 重试（详见 2.3） |
| ⑦ | export-results | 若决策为 `INSTALL_DEPENDENCY` 则暂停不导出；否则导出测试代码 + 报告 + 版本记录，并复制源文件到输出目录 |

### 2.3 自愈循环机制

自愈循环是系统最核心的能力，位于 `runSelfHealing()` 函数。其逻辑如下：

```
[执行测试] → 通过? ─是→ [质量检查] → 合格? ─是→ [导出结果]
              │否                    │否
              └─────→ [AI 诊断根因] ←──────┘
                          │
                   [决策: 是否测试代码错误?]
                    是 │          │ 否
           [重新生成测试代码]    [报告用户/请求环境修复]
              (切换 Pro 模型)        → 终止自愈
                    │
              [重新执行测试] → 回到起点
```

**关键设计**：

1. **快慢双通道**：首次生成与决策类任务用 `deepseek-v4-flash`（速度优先）；自愈重试的测试代码重写与高质量用例设计用 `deepseek-v4-pro`（准确率优先）。
2. **两段式诊断**：先由 `diagnosisAgent` 产出自然语言诊断报告（直接写入最终报告），再由 `diagnosisDecisionAgent` 将其转换为结构化 JSON 决策。两段式拆分降低了单一 LLM 出错率。
3. **决策校正 `reconcileNextAction()`**：以 `diagnosis_type` 为准重映射 `next_action`，纠正 LLM 倾向于无脑返回"重生成测试代码"的问题：
   - `TEST_CODE_ERROR` + 置信度 ≥ 0.7 → 重生成测试代码（继续自愈）
   - `SOURCE_RUNTIME_ERROR` / `BEHAVIOR_MISMATCH` → 报告用户（自愈不修源码）
   - `ENVIRONMENT_ERROR` → 安装依赖（暂停导出）
   - 其他/置信不足 → 请求用户确认
4. **跳出条件**：诊断类型非 `TEST_CODE_ERROR` 或置信度低于 0.7 时，立即停止自愈并 finalize，避免无效重试。
5. **版本管理**：每次重生成自动记录到 `versions[]`，最终报告保留完整版本历史，便于追溯。

### 2.4 多 Agent 协作架构

系统在 `src/mastra/agents/` 下定义了 7 个 Agent（6 个注册进 Mastra，`diagnosisDecisionAgent` 由工作流内部直接调用）：

| Agent | 导出名 | 职责 | 模型通道 |
| :--- | :--- | :--- | :--- |
| test-case-agent | `testCaseAgent` | 根据源代码结构设计测试用例（含边界值、异常路径） | flash |
| test-code-agent | `testCodeAgent` | 将用例翻译为可执行测试代码（首次生成） | flash |
| test-code-agent | `testCodeAgentPro` | 自愈重试时重写测试代码，精度更高 | pro |
| diagnosis-agent | `diagnosisAgent` | 分析测试失败根因，输出自然语言诊断 | flash |
| diagnosis-agent | `diagnosisAgentPro` | 自愈重试时的高精度诊断 | pro |
| diagnosis-agent | `diagnosisDecisionAgent` | 将自然语言诊断转为结构化决策（JSON） | flash |
| cli-conversation-agent | `cliAgent` | 理解用户自然语言需求、制定执行计划 | flash |

**Agent Prompt 设计要点**（贯穿反迎合理念）：

- **testCaseAgent**：3 条硬禁令（禁止回避异常边界输入、禁止按"代码会输出什么"填 expected、禁止用无效输入替换危险输入）；强制覆盖输入清单（数值 0/MAX/MIN、字符串 null/空/特殊字符、除数为 0 等）。
- **testCodeAgent / Pro**：3 条硬禁令（禁止 `assertFalse→assertThrows` 迎合、禁止换输入绕 bug、禁止改 expected 匹配错误输出）；强制末尾输出 `SUSPICIOUS_BEHAVIORS` JSON 注释块记录可疑行为，作为抓 bug 评测依据。
- **diagnosisAgent**：输出自然语言（非 JSON），聚焦根因分类（源代码缺陷/测试代码错误/环境缺失），只引用关键报错行。

## 3. 核心模块实现

### 3.1 工具层（Tools）

系统在 `src/mastra/tools/` 下定义了 7 个工具，LLM 通过工具调用完成实际执行：

| 工具 | 关键函数 | 输入 → 输出 | 实现要点 |
| :--- | :--- | :--- | :--- |
| parse-source-code | `parseSourceCode()` | 源码+文件名+语言 → 符号表 | Python 走 `ast`；Java/C++ 走 Tree-sitter，提取类、方法签名、参数类型、起止行号 |
| execute-tests | `executePytest()` / `executeGeneratedTests()` | 测试代码+源码+超时 → 执行结果 | Python 经子进程隔离执行；C++ 委托适配器；默认超时 60s |
| measure-coverage | `measureCoverage()` | 测试+源码+语言 → 覆盖率 | 三套实现：Python coverage.py / Java JaCoCo（解析 jacoco.xml）/ C++ gcov；失败降级到符号覆盖率 |
| export-cases | `exportCases()` | 用例+代码+上下文 → 导出文件列表 | 委托 Python 脚本生成 Markdown 报告；含条件审批（输出目录在 CWD 外才弹 y/n） |
| read-file | `readFile()` | 路径 → 内容+元信息 | `fs.readFile` + 行数统计 |
| write-file | `writeFile()` | 路径+内容 → 写入结果 | 自动创建父目录；强制审批 + 风险自评字段 |
| logger | `writeLog()` | 日志事件 → JSONL 落盘 | 写入 `output/logs/agent.log`，含 LLM 请求/响应/重试、工作流步骤、工具调用等事件 |

### 3.2 语言适配器层（Adapters）

系统通过 `LanguageAdapter` 接口将语言差异封装在适配器内，工作流只面向统一类型，实现"一次编排、三语通用"。

**统一接口契约**（`src/mastra/languages/types.ts`）：

```typescript
interface LanguageAdapter {
  language: string;           // 语言标识
  displayName: string;        // 显示名称
  extensions: string[];       // 支持的扩展名
  testFramework: string;      // 测试框架
  codeFence: string;          // 代码围栏语言
  parseSource(code): SourceAnalysis;           // 解析源码为符号表
  buildGenerationContext(analysis): string;    // 构造 LLM 上下文
  executeTests(opts): ExecutionResult;         // 在临时工作目录中隔离执行测试
  diagnose(result): Diagnosis;                 // 正则规则快速诊断
  exportArtifacts(opts): ExportResult;         // 导出产物
}
```

**三语言实现差异**：

| 语言 | 解析方式 | 执行方式 | 覆盖率工具 | 诊断方式 |
| :--- | :--- | :--- | :--- | :--- |
| Python | 标准库 `ast`（确定性提取） | pytest 子进程隔离 | coverage.py | 正则规则引擎 |
| Java | Tree-sitter-java | 自建临时 Maven 工程，`mvn verify` | JaCoCo Maven 插件 | 正则规则引擎 |
| C++ | Tree-sitter-cpp | g++ 编译 GoogleTest 可执行文件 | gcov（预留） | 正则规则引擎 |

**关键实现细节**：

- **Python 适配器**：`buildGenerationContext` 强调"必须从运行时模块名导入，不要 import source"；`toPythonModuleName()` 将文件名转为合法 Python 模块名。
- **Java 适配器**：`executeJavaTests` 自动生成含 JUnit5 + jacoco 插件的 `pom.xml`，`spawnSync` 执行 `mvn verify`，`parseSurefireReports()` 解析 surefire XML 测试报告。
- **C++ 适配器**：`resolveGtestCompileConfig()` 优先用 `GTEST_ROOT` 源码编译，否则 `-lgtest -lgtest_main`；Windows 上 `resolveToolchainBinDirs()` 解析 MSYS2 UCRT64 工具链路径并前置到 PATH。

### 3.3 运行时基础设施（Runtime）

#### Python 桥接（`python-bridge.ts`）

所有 Python 工具（解析、执行、导出、覆盖率）的统一子进程通道：

```typescript
callPythonScript<T>(scriptName, inputData, timeoutMs): PythonCallResult<T>
```

- 通过 `spawnSync("python", [scriptPath])` 启动子进程，stdin 传 JSON 参数，stdout 回 JSON 结果。
- `resolvePythonRuntimeDir()` 智能定位 `python-runtime/` 目录（优先环境变量，否则从 cwd/`__dirname`/`INIT_CWD` 向上查找，须含全部 4 个脚本）。
- 统一设置 `PYTHONIOENCODING=utf-8`、`PYTHONUTF8=1` 保证跨平台编码一致。

#### 命令执行器（`command-runner.ts`）

- `assessCommandRisk()`：正则匹配命令风险等级（high/medium/low），`rm -rf`/`format`/`git reset --hard` 判为 high，`npm install`/`pip install` 判为 medium。
- `runCommandInVisibleTerminal()`：Windows 专用，生成临时 `.ps1` 脚本并弹出可见终端窗口执行命令，供 `INSTALL_DEPENDENCY` 场景下让用户可见地执行环境命令。

#### 环境与 LLM 守卫（`env.ts`）

- `loadProjectEnv()`：从 cwd 向上查找 `.env` 并加载，支持 `${VAR}` 变量展开。
- `assertLlmAvailable(stage)`：所有 LLM 调用前的统一守卫，校验 API Key 有效性、`TESTGENERATE_LLM` 开关、Key 是否含非 ASCII 或占位符。

#### CLI 输出（`cli-output.ts`）

提供显式彩色输出 API（`logAgent`/`logInfo`/`logWarn`/`renderProgressBar` 等），检测 `NO_COLOR`/`CI`/非 TTY 任一条件即降级为无色文本，管道/重定向/CI 环境不产生 ANSI 乱码。

## 4. Web 系统实现

### 4.1 后端架构（`src/server/`）

后端基于 Express 框架，采用经典的 Controller-Service-Route 分层：

| 目录 | 职责 | 主要模块 |
| :--- | :--- | :--- |
| config/ | 数据库与环境配置 | database.ts（Prisma Client 单例）、env.ts |
| controllers/ | 请求处理 | auth / session / task / file / workspace / apiKey / config |
| services/ | 业务逻辑 | auth / session / task / file / workspace / apiKey / llm / stream |
| routes/ | 路由定义 | auth / session / task / file / workspace / apiKey / config / stream |
| middleware/ | 中间件 | auth（JWT 鉴权）、validator（参数校验）、errorHandler（错误处理） |
| utils/ | 工具函数 | jwt、crypto（bcrypt 加密）、response（统一响应） |

**应用装配**（`app.ts`）：

- 安全中间件：`helmet()` 设置安全响应头，`cors` 配置跨域（开发环境允许所有来源）。
- 请求解析：`cookieParser()` + `express.json({ limit: '10mb' })`。
- 请求日志：记录每个请求的方法、路径、状态码、耗时。
- 路由挂载：统一前缀 `/api/v1`，含健康检查端点 `/api/v1/health`。
- 错误处理：404 处理器 + 全局错误处理器。

**统一响应格式**（`utils/response.ts`）：

```typescript
interface ApiResponse<T> {
  code: number;        // 0 表示成功，非 0 为错误码
  message: string;
  data?: T;
  details?: any;
  timestamp: number;
}
```

- 内置 `ErrorCode` 枚举（认证 1xxx / API Key 2xxx / 任务 3xxx / 工作空间 4xxx / 文件 5xxx / 会话 6xxx / 系统 9xxx），每个错误码映射到对应 HTTP 状态码。
- `safeJsonStringify()` 处理 BigInt 序列化问题（数据库主键为 BigInt）。
- 提供 `sendSuccess` / `sendPaginated` / `sendError` / `sendValidationError` 四类响应函数。

### 4.2 执行器与流式推送

Web 端通过两个执行器接入核心能力，并通过 SSE 流式推送进度：

- **workflow-executor.ts**：接入 7 步工作流，支持 `onLog`/`onStep` 回调实时上报进度。
- **agent-executor.ts**：接入自主 Agent，记录所有工具调用，支持超时与最大步数限制。
- **stream.service.ts**：真正调用 autonomous agent 与 workflow，通过 SSE 流式输出每个步骤到前端。支持 `onProgress`/`onComplete`/`onIncomplete`/`onError`/`onAsk` 回调；当 Agent 调用 `ask-user` 工具时，通过 `onAsk` 弹出对话框收集用户输入后重发。
- **task-runtime-registry.ts**：任务运行时注册表，支持任务取消（`cancelTaskRun`/`cancelSessionTaskRuns`），通过 `throwIfTaskRunCancelled()` 在工作流关键节点检查取消标志。

### 4.3 数据持久化（Prisma + MySQL）

数据库包含 10 张表，覆盖用户、鉴权、会话、任务、文件全链路：

| 表名 | 说明 | 关键字段 |
| :--- | :--- | :--- |
| users | 用户 | id, username, email, passwordHash, role, status |
| api_keys | API 密钥 | userId, keyHash, prefix, permissions, rateLimit, expiresAt |
| workspaces | 工作空间 | userId, name, basePath, isDefault |
| sessions | 会话 | userId, workspaceId, title, mode, messageCount, totalTokens |
| messages | 消息 | sessionId, role, content, messageType, parentId |
| tasks | 任务 | userId, taskId, status, mode, sourceFile, language, result |
| task_logs | 任务日志 | taskId, level, step, message |
| uploaded_files | 上传文件 | userId, filename, path, hash, purpose |
| file_contents | 文件内容 | fileId, chunkIndex, content（分块存储） |
| usage_stats | 使用统计 | userId, date, taskCount, tokenUsage |

**枚举设计**：`Role`（user/admin/super_admin）、`TaskStatus`（pending/running/completed/failed/cancelled）、`TaskMode`（workflow/autonomous）、`MessageRole`（user/assistant/system/tool）等，约束字段取值。

### 4.4 前端架构（`client/src/`）

前端基于 React + Vite + Ant Design，采用页面+组件+API+Store 分层：

| 目录 | 内容 |
| :--- | :--- |
| pages/ | Dashboard（仪表盘）、Chat（对话）、Sessions（会话历史）、Files（文件管理）、Workspaces（工作空间）、ApiKeys（密钥管理）、Auth（登录/注册）、Profile（个人中心）、Settings（LLM 配置） |
| components/ | Chat/ChatRightPanel、Common/CodeViewer、CodeViewerModal、MarkdownMessage、Layout/MainLayout |
| api/ | client（axios 封装）、auth、sessions、files、workspaces、apiKeys |
| stores/ | authStore、sessionStore（Zustand 状态管理） |

前端支持 **Agent 模式**与 **Workflow 模式**两种对话方式，通过 SSE 实时接收任务进度与结果，支持代码查看、Markdown 渲染、文件预览。

## 5. 自主 Agent 模式

除工作流驱动模式外，系统提供自主 Agent REPL 模式（`src/autonomous/`），LLM 自主规划、自主调用工具，用户在关键步骤做 y/n 审批：

| 模块 | 职责 |
| :--- | :--- |
| autonomous-agent.ts | 自主 Agent 主体，集成 9 个工具 |
| autonomous-loop.ts | REPL 多轮对话循环 |
| approval.ts | 工具调用审批策略 |
| safety.ts | 命令风险评级 |
| shell-runner.ts | shell 命令执行 |
| ask-user-tool.ts | 主动向用户提问 |

该模式下 Agent 可读取文件、写文件、执行 shell 命令、调用测试生成工具，所有高风险操作经审批策略过滤后执行，兼顾灵活性与安全性。

## 6. 关键技术实现

### 6.1 JSON 鲁棒性处理

针对 LLM 输出 JSON 不稳定的问题，系统实现多重保障：

- `parseJsonValue()` + `findMatchingJsonEnd()`：基于括号配对的容错解析。
- 截断续写：`continueTestCaseJson()` 对被截断的 JSON 做续写与重叠拼接。
- LLM 修复：解析失败时调用 LLM 修复无效 JSON。
- Zod 校验：工具入参出参严格校验，不通过则重试。

### 6.2 覆盖率测量统一入口

`src/mastra/tools/coverage-tool.ts` 暴露 `measureCoverage()` 作为三语言覆盖率统一入口，工作流在自愈循环的多个节点自动调用。失败时优雅降级到符号覆盖率 `calculateCoverage()`，并在报告中标注 `coverage_tool` 字段，保证覆盖率数据不缺失。

### 6.3 LLM 重试与降级

- `withLlmRetries()`：通用 LLM 重试包装，支持配置重试次数（默认 2）。
- `setLlmRetriesExhaustedHandler()`：重试耗尽时的钩子，允许 CLI 追加人工重试。
- 快慢双通道：首生成用 flash（速度优先），自愈重试用 pro（准确率优先）。

### 6.4 风险分级审批（V2.4）

- write-file 工具强制审批。
- export-cases 工具条件审批（输出目录在 CWD 外才审批）。
- command-runner 命令风险评级（high/medium/low）。
- 工具含 `risk` 自评字段，配合审批策略实现安全控制。
