> **文档版本**：V1.0

> **适用版本**：V3.1

## 1. 系统简介

TestGenerate Agent 是一个智能测试生成助手，能读懂你的源代码，自动设计测试用例、生成可执行的测试代码、真实运行测试、并在失败时自动修复，最终交付测试代码与详细报告。支持 Python、Java、C++ 三种语言。

你可以通过两种方式使用它：

- **命令行（CLI）**：在终端输入命令或像聊天一样交互，适合本地快速生成。
- **网页端（Web）**：浏览器图形界面，适合管理历史、上传文件、团队协作。

本手册面向最终用户，指导你完成日常使用。

## 2. 快速开始

### 2.1 我该用哪种方式？

| 你的场景 | 推荐方式 |
| :--- | :--- |
| 一次性给单个文件生成测试 | CLI 命令行模式 |
| 想用自然语言描述需求、边聊边生成 | CLI 交互模式 |
| 让 AI 自己规划、自主调用工具 | CLI 自主模式 |
| 需要管理多个项目、查看历史、团队共享 | Web 网页端 |

### 2.2 三十秒上手（CLI）

```bash
# 给一个 Python 文件生成测试
npm run generate -- --input ./我的代码.py --output ./我的测试

# 生成完成后，到输出目录查看结果
ls ./我的测试
```

你会得到：测试代码文件 + 一份 Markdown 测试报告。

### 2.2 三十秒上手（Web）

1. 浏览器打开 `http://localhost:5173`。
2. 注册账号并登录。
3. 上传你的源代码文件。
4. 新建会话，选择"工作流模式"，发送消息"帮我生成测试"。
5. 等待进度条走完，查看生成的测试与报告。

## 3. CLI 使用指南

### 3.1 三种模式

#### 模式一：命令行模式（最直接）

一次性给定输入文件，自动跑完全流程。

```bash
npm run generate -- --input <源文件路径> [--output <输出目录>] [其他选项]
```

示例：

```bash
# Python
npm run generate -- --input ./calc.py --output ./tests

# Java（自动检测）
npm run generate -- --input ./UserService.java

# C++，显式指定语言
npm run generate -- --input ./vector.tcc --language cpp

# 附加需求："重点测试边界值"
npm run generate -- --input ./calc.py --requirements "重点测试边界值和异常输入"
```

#### 模式二：交互模式（像聊天）

进入后用自然语言描述需求，系统会理解并执行。

```bash
npm run generate -- --interactive
```

进入后可以这样说：

```
> 帮我测试 ./src/calculator.py
> 这个函数有除法，注意除以零的情况
> 输出到 ./my-tests 目录
```

#### 模式三：自主模式（AI 自己规划）

AI 自主决定调用哪些工具、如何推进，遇到高风险操作会请你 y/n 审批。

```bash
npm run generate -- --autonomous
```

适合复杂场景：多文件、需要执行 shell 命令、需要中途调整策略。

### 3.2 命令选项速查

| 选项 | 简写 | 说明 | 默认值 |
| :--- | :--- | :--- | :--- |
| `--input` | `-i` | 源文件路径（.py/.java/.cpp/.cc/.hpp/.h） | 必填（命令行模式） |
| `--output` | `-o` | 输出目录 | `./output/exports` |
| `--language` | `-l` | 语言：python/java/cpp | auto（自动检测） |
| `--max-attempts` | - | 最大自愈尝试次数 | 3 |
| `--llm-retries` | - | 每次 LLM 请求重试次数 | 2 |
| `--requirements` | - | 额外需求文本 | - |
| `--requirements-file` | - | 从文件读取额外需求 | - |
| `--interactive` | `-I` | 自然语言交互模式 | 关闭 |
| `--autonomous` | `--agent` | 自主 Agent REPL 模式 | 关闭 |
| `--help` | `-h` | 显示帮助 | - |

### 3.3 看懂输出

执行完成后，输出目录包含：

```
my-tests/
├── test_xxx.py              ← 可执行的测试代码
├── test_cases.md            ← 测试用例清单（设计阶段）
├── test_cases_plan.md       ← 测试计划（预审版）
├── report_xxx.md            ← 完整测试报告（含覆盖率、诊断、版本历史）
└── xxx.py                   ← 被测源文件副本
```

**测试报告示例片段**：

```markdown
## 总计
- 测试用例：12 个
- 通过：10 个
- 失败：2 个

## 覆盖率
- 真实行覆盖率：85.5%（工具：coverage.py）
- 分支覆盖率：70.0%

## 缺陷诊断
removeOddNumbers 存在 erase-in-loop 索引跳跃缺陷...
```

### 3.4 CLI 彩色输出说明

终端中会用颜色区分信息：亮黄色是 Agent 思考，青色是用户提示符，绿色是成功，红色是错误。如果你把输出重定向到文件或在 CI 环境，会自动变成无色纯文本，不会出现乱码。

## 4. Web 使用指南

### 4.1 注册与登录

1. 访问系统首页，点击"注册"。
2. 填写用户名、邮箱、密码（密码需有一定强度）。
3. 注册成功后用账号密码登录，系统签发 JWT 令牌（默认 2 小时有效）。

### 4.2 仪表盘

登录后进入仪表盘，展示：

- 统计概览（任务数、消息数、Token 用量）
- 快速操作入口
- 最近活动

### 4.3 工作空间管理

工作空间是你的项目工作目录，用于组织会话与文件。

1. 进入"工作空间"页面，点击新建。
2. 填写名称与基础路径（Base Path，指向本地项目目录）。
3. 可设为默认工作空间。
4. 通过"目录浏览器"浏览工作空间下的文件并选择。

### 4.4 文件管理

1. 进入"文件"页面，点击上传。
2. 选择源代码文件（支持 .py/.java/.cpp 等）。
3. 文件入库后可在会话中引用。
4. 单文件上限 10MB。

### 4.5 会话与对话

这是核心使用页面，支持两种模式：

#### 工作流模式（Workflow）

固定 7 步流水线，适合标准测试生成：

1. 新建会话，模式选"工作流"。
2. 上传或选择源文件。
3. 在对话框输入需求（如"生成测试，关注边界值"）。
4. 发送后，右侧进度面板实时显示 7 步进度。
5. 完成后展示测试代码与报告，可在线查看、下载。

#### 自主模式（Autonomous）

AI 自主规划与工具调用，适合复杂场景：

1. 会话模式选"自主"。
2. 用自然语言描述任务。
3. AI 会逐步思考、调用工具，过程实时展示。
4. 当 AI 需要你确认或输入时，会弹出对话框，回答后继续。

### 4.6 会话历史

- 所有会话自动保存，可在"会话"页面回溯。
- 支持按状态筛选（活跃/已归档）。
- 点击历史会话可查看完整对话与任务结果。

### 4.7 任务管理

- 每次生成会创建一个任务，状态包括：待处理/运行中/已完成/失败/已取消。
- 任务页可查看执行时长、尝试次数、Token 用量。
- 运行中的任务可取消。
- 任务日志记录每一步详情。

### 4.8 API Key 管理

如果你需要通过 API 集成系统：

1. 进入"API Key"页面，创建密钥。
2. 系统返回密钥前缀（完整密钥仅显示一次，请保存）。
3. 可配置权限（read/generate）、速率限制、过期时间。
4. 调用 API 时在请求头携带 `X-API-Key`。

### 4.9 LLM 配置

在"设置 → LLM 配置"页面：

- 配置 DeepSeek API Key。
- 调整模型通道（flash/pro）。
- 配置生效后所有会话使用新设置。

### 4.10 个人中心

- 查看账号信息与角色。
- 修改头像、偏好设置。
- 查看使用统计。

## 5. 三语言使用要点

### 5.1 Python

- 框架：pytest
- 覆盖率：coverage.py（真实行/分支覆盖率）
- 注意：系统会自动处理模块导入，你无需关心 import 路径。
- 需本机已安装 pytest。

### 5.2 Java

- 框架：JUnit 5 + Maven
- 覆盖率：JaCoCo
- 注意：系统自动生成 Maven 工程（pom.xml），需本机有 `mvn` 与 JDK 17+。
- 首次执行 Maven 会下载依赖，需联网。

### 5.3 C++

- 框架：GoogleTest
- 覆盖率：gcov（gcov 已实现，OpenCppCoverage 待接入）
- 注意：需本机 g++ 支持 C++17，推荐 MSYS2 UCRT64 工具链。
- 可设置 `GTEST_ROOT` 指定 GoogleTest 源码路径。

## 6. 理解测试结果

### 6.1 通过 vs 失败

- **通过**：测试断言全部成立，说明被测代码在该场景下行为符合预期。
- **失败**：测试断言不成立——这**不一定是坏事**。系统设计的初衷就是暴露缺陷，失败用例往往揭示了源代码的真实 bug。

### 6.2 诊断类型

测试失败时，系统会诊断根因并分类：

| 诊断类型 | 含义 | 系统行为 |
| :--- | :--- | :--- |
| TEST_CODE_ERROR | 测试代码本身有错（语法、import） | 自动重新生成测试代码（自愈） |
| BEHAVIOR_MISMATCH | 行为不符合预期（疑似源码缺陷） | 报告缺陷，停止自愈 |
| SOURCE_RUNTIME_ERROR | 源代码运行时崩溃 | 报告用户，不自愈 |
| ENVIRONMENT_ERROR | 环境缺失（如未装依赖） | 提示安装依赖，暂停导出 |

### 6.3 覆盖率解读

| 指标 | 含义 |
| :--- | :--- |
| 真实行覆盖率 | 被执行的代码行占比（coverage.py/JaCoCo/gcov 测量） |
| 分支覆盖率 | 分支判断被覆盖的占比 |
| 符号覆盖率 | 被至少一个用例覆盖的函数/方法占比（降级方案） |

若真实覆盖率工具失败，系统会降级到符号覆盖率并在报告中标注。

### 6.4 版本历史

自愈循环中每次重新生成测试都会记录版本。报告中保留完整版本历史，你可以对比不同版本的测试代码，理解 AI 的修复思路。

## 7. 常见使用场景

### 场景一：给单个函数生成边界测试

```bash
npm run generate -- --input ./math.py --requirements "重点测试边界值：0、最大值、最小值、负数、除以零"
```

### 场景二：让 AI 自己决定怎么测

```bash
npm run generate -- --autonomous
# 进入后说：分析 ./src/ 目录下的所有 .py 文件，给每个文件生成测试
```

### 场景三：团队协作管理测试资产

- 在 Web 端创建工作空间，指定项目目录。
- 上传关键源文件。
- 多人共享会话历史，复用已生成的测试。

### 场景四：测试一个已知有 bug 的函数

直接提交，系统会生成能暴露该 bug 的测试用例（反迎合设计保证不会绕过缺陷），报告中会给出根因分析与修复建议。

## 8. 常见问题

**Q1：生成的测试全部通过，但我确定代码有 bug，怎么办？**
A：检查是否触发了"反迎合"——如果测试用 `assertThrows` 接住了本应暴露的异常，或在 expected 里填了错误值，可在需求中强调"不要迎合源代码行为，期望值应基于规范而非实际输出"。也可提高 `--max-attempts` 让系统多尝试。

**Q2：自愈循环一直不收敛怎么办？**
A：可能是源代码本身有运行时缺陷（BEHAVIOR_MISMATCH/SOURCE_RUNTIME_ERROR），系统会正确停止自愈并报告。请查看报告中的诊断类型。

**Q3：提示环境错误（ENVIRONMENT_ERROR）怎么办？**
A：按提示安装缺失依赖（如 pytest、Maven、gtest）。CLI 自主模式下，系统会弹出可见终端让你确认执行安装命令。

**Q4：C++ 测试编译失败？**
A：确认 g++ 支持 C++17，推荐 MSYS2 UCRT64。也可设置 `GTEST_ROOT` 环境变量指向 GoogleTest 源码。

**Q5：Web 端任务卡住不动？**
A：可能是 LLM 响应慢或网络问题。可在任务页取消后重试。若反复失败，检查 LLM 配置页的 API Key 是否有效。

**Q6：能测多文件项目吗？**
A：CLI 命令行/工作流模式针对单文件；多文件项目建议用自主模式，让 AI 自主规划读取多个文件并分别生成测试。

**Q7：生成的测试代码能直接放进我的项目吗？**
A：可以。输出目录的测试代码是完整可执行的，复制到你的项目对应测试目录即可。Java 产物含完整 Maven 工程结构，C++ 产物可直接 g++ 编译。

## 9. 最佳实践

1. **写清需求**：在 `--requirements` 或对话中明确测试重点（边界、异常、性能），生成质量更高。
2. **关注失败用例**：失败用例往往是真实的代码缺陷，别急着认定是测试问题，先看诊断类型。
3. **善用自愈**：适当调高 `--max-attempts`（如 5）能让系统多尝试修复测试代码错误。
4. **核对覆盖率**：覆盖率低的函数说明测试不足，可针对这些函数追加需求重新生成。
5. **Web 端管理历史**：长期项目用 Web 端，便于追溯每次生成的测试与报告。
6. **生产环境改密钥**：部署 Web 端时务必修改 `JWT_SECRET` 与数据库密码，启用限流。
