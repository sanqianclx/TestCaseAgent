## 1 项目概述

### 1.1 项目背景

在软件开发生命周期中，测试用例设计和测试代码编写是保障软件质量的核心环节。然而，传统的人工方式面临四大痛点：

1. **效率低下**：开发工程师需逐条分析代码逻辑并手工编写测试用例和测试代码，耗时巨大
2. **质量参差**：不同工程师编写的测试用例粒度和测试代码覆盖率存在显著差异
3. **维护困难**：代码频繁变更时测试代码更新滞后
4. **调试低效**：测试失败时需要人工排查是源代码Bug还是测试代码自身的问题

为应对上述挑战，本系统研发一款基于大语言模型（LLM）的智能测试用例生成 Agent，覆盖 Python、Java、C++ 三类主流语言，通过统一的 Language Adapter 接口屏蔽语言差异，共享同一套解析、用例生成、测试代码生成、执行、覆盖率采集、失败诊断与自愈循环主流程。

### 1.2 项目定位

本项目实现的是一个具备自主决策能力的智能体（Agent），核心差异化特征：

1. **自主决策**：Agent根据输入内容和当前上下文，自主规划执行路径
2. **工具调用**：Agent通过调用预定义的工具集 API 完成具体操作，LLM 负责决策"何时调用哪个工具"
3. **多语言统一接口**：所有语言相关能力（解析、执行、覆盖率、导出）由 `Language Adapter` 模式封装，Agent 调度时不感知语言差异
4. **动态工作流**：每次执行路径可能不同，取决于源代码复杂度、目标语言和测试结果
5. **循环运行与自愈**：Agent在循环中持续运行。测试失败时输出结构化诊断；测试代码错误则自动重生成并再次执行，源代码运行错误或行为不一致则向用户报告证据并等待确认

核心工具集（V2.1 起）：

- `read-file` / `parse-source-code`：按 `language` 字段路由到对应 Language Adapter
- `execute-tests`：按语言分发到 python-adapter / java-adapter / cpp-adapter
- `measure-coverage`：Python 走 coverage.py，Java 走 JaCoCo，C++ 走 gcov
- `write-file` / `export-cases`：按语言输出对应后缀的测试代码 + Markdown 文档
- `create-test-code-version`：保存每轮生成或自愈后的测试代码版本
- `diagnose-failure`：由 diagnosis-agent 承担，输出诊断类型、置信度、证据和建议动作

### 1.4 核心术语

| 术语                 | 定义                                                                                           |
| ------------------ | -------------------------------------------------------------------------------------------- |
| Agent（智能体）         | LLM + 工具集 + 自主调用机制构成的自治系统                                                                    |
| Tool（工具）           | 可供Agent调用的函数API，每个工具有明确的输入输出规范                                                               |
| Tool Call（工具调用）    | LLM在推理过程中决定调用某个工具并传入参数的行为                                                                    |
| Run Loop（运行循环）     | Agent持续执行的"感知→思考→行动→观察"循环                                                                    |
| Self-Healing（自愈循环） | 测试失败后自动判断根因、修复测试代码、重新执行的闭环                                                                   |
| Guardrail（护栏）      | 在高风险操作执行前进行的安全检查机制                                                                           |
| Test Code（测试代码）    | Agent生成的、可直接被测试框架执行的单元测试代码                                                                   |
| AST（抽象语法树）         | 代码结构解析结果。Python 走标准库 `ast`；Java / C++ 走 `tree-sitter` 解析得到的结构，用于稳定提取类、方法、参数、注解、namespace 与行号 |
| Language Adapter   | 按语言封装解析、执行、覆盖率、导出能力的统一接口，python-adapter / java-adapter / cpp-adapter 各自实现                    |
| Language Registry  | 根据文件扩展名或 `--language` 参数路由到对应 Language Adapter 的注册表                                          |
| InMemoryStore      | 进程内会话期中间结果（解析中间产物、覆盖率原始报告、自愈诊断上下文）保存介质，进程结束即清理                                               |
| Coverage Tool      | 对应语言的真实覆盖率采集工具：Python `coverage.py`、Java `JaCoCo`、C++ `gcov`                                 |
| Diagnosis（失败诊断）    | 对测试失败原因的结构化判断，包含类型、置信度、证据和建议动作                                                               |
| Mastra             | TypeScript AI Agent框架，用于实现Agent、Tools、Workflows、Guardrails和可观测调试能力                           |
| Workflow（工作流）      | Mastra中用于定义明确多步骤流程的编排机制，适合解析、生成、执行、诊断、自愈这类固定主链路                                              |

### 2.3 代码解析

**需求描述**：Agent通过 `parse-source-code` 工具对上传的源代码文件进行结构化解析，按 `language` 字段路由到对应 Language Adapter：

- 模块名称和导入列表（Python `import / from ... import`、Java `import`、C++ `#include`）
- 类定义：类名、继承关系、类注释
- 方法/函数定义：函数名、参数列表（名称、类型注解、默认值）、返回类型注解、类注释、起始行号和结束行号
- Java 额外字段：包名（`package`）、注解（`@Override` 等）、modifier（`public`/`static`/`final` 等）
- C++ 额外字段：namespace、模板参数（`template`）、include 列表
- 函数逻辑摘要（V1.0 可选，V2.0 起使用 LLM 生成）

**解析输出格式**（JSON结构）：

```json
{
  "module_name": "user_service",
  "imports": ["from flask import ..."],
  "classes": [
    {
      "name": "UserService",
      "bases": ["BaseService"],
      "docstring": "...",
      "methods": [
        {
          "name": "register_user",
          "params": [{"name": "phone", "type": "str", "default": null}],
          "return_type": "bool",
          "docstring": "...",
          "start_line": 15,
          "end_line": 42
        }
      ]
    }
  ],
  "functions": [...]
}
```

**边界情况**：

- 源文件存在语法错误时，抛出结构化错误，包含错误行号和原因，不进行后续生成
- 文件扩展名与 `--language` 不匹配时（如 `.py` 配 `--language=java`），抛出 `FILE_TYPE_UNSUPPORTED`，不进入解析
- 源文件中的 import 解析结果仅用于生成 mock/fixture 建议，不影响 AST 解析过程
- 空文件或仅含注释和 import 的文件：输出空结构，明确提示
- 大型文件（超过 500 行或 20 个函数）：输出文件规模警告，LLM 按模块/类分批处理

### 2.4 测试生成

Agent根据AST解析结果和LLM逻辑摘要生成两部分内容：

**A. 测试用例文档**：用例编号、标题、优先级（P0/P1/P2/P3）、用例类型（功能/边界/异常）、前置条件、测试步骤、预期结果、关联方法。

**B. 测试代码**：可被对应语言测试框架直接执行的单元测试代码。Language Adapter 根据源语言生成对应后缀的测试代码：

- Python → pytest 测试代码（`.py`，`def test_xxx`）
- Java → JUnit 5 测试代码（`.java`，`@Test` + `assertEquals`/`assertThrows`）
- C++ → GoogleTest 测试代码（`.cpp`，`TEST() / ASSERT_EQ() / EXPECT_THROW()`）

示例（Python）：

```python
def test_register_user_success():
    result = register_user("13800001111", "test_user", "Pass1234", "123456")
    assert result == True

def test_register_user_duplicate():
    register_user("13800001111", "user1", "Pass1234", "123456")
    result = register_user("13800001111", "user2", "Pass5678", "654321")
    assert result == False
```

**生成策略**：功能测试（正常流程和分支路径）、边界测试（等价类划分和边界值分析）、异常测试（空值、超长输入、特殊字符、类型错误）。

### 2.5 测试执行与自愈

Agent调用 `execute-tests` 在临时执行目录中执行测试代码，Language Adapter 按源语言分发到 python-adapter / java-adapter / cpp-adapter。执行结果包含：通过用例数、失败用例数、错误用例数、标准输出、标准错误、退出码、耗时、是否超时，以及逐用例结果列表（`test_results`）。

若测试失败，Agent调用 `diagnose-failure` 分析根因：

**情况A：`TEST_CODE_ERROR`**。Agent保存当前测试代码版本，重新生成下一版本测试代码、写入文件、再次执行。此过程最多自动重试 3 轮。

**情况B：`SOURCE_RUNTIME_ERROR`**。Agent输出错误报告：错误所在源文件和行号、异常类型和具体原因、按语言分写的复现命令、traceback 证据：

- Python：`pytest -q <test_file> -k <test_name>`
- Java：`mvn -q -Dtest=<ClassName>#<testName> test`
- C++：`./<gtest_executable> --gtest_filter=<TestSuite>.<TestName>`

**情况C：`BEHAVIOR_MISMATCH`**。Agent输出实际结果与预期结果的差异，说明预期来自需求文本、类注释、函数名推断或 LLM 推断，不自动修改源代码。

**情况D：`UNKNOWN`**。Agent输出诊断证据和建议用户确认的问题，不进入自愈循环。

## 3 业务流程与运行机制

### 3.1 主业务流程

1. 用户通过 CLI 传入源文件路径或粘贴代码，Language Registry 根据扩展名或 `--language` 参数路由到对应 Language Adapter；Web 阶段也可通过页面上传文件或粘贴代码。
2. 系统读取源代码并进行路径、类型和基础大小检查。
3. Agent 调用 `parse-source-code` 工具提取模块、类、函数、方法、参数、返回注解、类注释、行号和 import/include 列表（Java 额外记录注解和 modifier，C++ 额外记录 namespace）。
4. Agent 调用 LLM 生成逻辑摘要、输入约束、边界条件和测试关注点。
5. Agent 生成测试用例文档，并据此生成对应语言测试代码（Python 走 pytest，Java 走 JUnit 5，C++ 走 GoogleTest）。
6. 系统保存测试代码版本，将代码写入临时执行目录和输出目录。
7. 系统按语言分发执行：Python `pytest`、Java `mvn verify`、C++ `g++` + gtest 二进制。采集执行结果（含 `test_results` 逐用例列表）。
8. 测试通过后调用 `measure-coverage` 采集真实行/分支覆盖率（Python coverage.py / Java JaCoCo / C++ gcov）。
9. 质量通过则导出测试用例、测试代码、执行报告与覆盖率；测试失败时进入失败诊断；仅当诊断为测试代码问题且置信度达到阈值时自动自愈。
10. 达到退出条件后返回最终结果、中间结果或需要用户确认的问题。

### 3.2 失败诊断业务规则

| 诊断类型                   | 典型证据                                                                                                                                                    | 系统处理                                |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| `TEST_CODE_ERROR`      | Python：traceback 指向生成测试文件；fixture 未定义；mock 路径错误；调用不存在的函数或参数不匹配。Java：`BUILD FAILURE` 指向测试文件、Mockito 路径错、import 缺失。C++：linker error 指向 gtest 文件、gmock 路径错 | 当置信度不低于 0.70 时自动重生成测试代码             |
| `SOURCE_RUNTIME_ERROR` | Python：traceback 指向源代码文件；语法错误、运行时异常或依赖缺失。Java：`package xxx does not exist`、JaCoCo 数据缺失、`Maven is not available`。C++：编译错误指向源代码头文件、链接器缺库                  | 输出错误位置、异常原因、traceback 证据和按语言分写的复现命令 |
| `BEHAVIOR_MISMATCH`    | 测试可执行，但实际结果与需求文本、docstring 或用例预期不一致                                                                                                                     | 输出预期来源、实际结果和差异点，等待用户确认              |
| `UNKNOWN`              | 证据不足，无法可靠判断根因                                                                                                                                           | 输出已知证据和建议用户补充的信息，不自动修改              |

### 3.3 自愈边界

自愈仅处理测试代码自身问题，不修改用户源代码。系统最多连续自愈 3 轮（`max-heal-attempts`），Agent 总运行轮次最多 20 轮（`max-rounds`）。若连续 3 轮仍失败，或诊断置信度不足，系统必须停止自动修改并报告当前版本、失败原因和人工介入建议。

### 3.4 退出条件

| 编号   | 退出条件         | 用户可见结果                                                                                                                                                        |
| ---- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| EC-1 | 测试通过且覆盖率采集完成 | 导出文件、测试代码、执行摘要、真实行/分支覆盖率（line\_rate / branch\_rate / per\_file / report\_path）                                                                                |
| EC-2 | 源代码运行错误      | 错误位置、原因、证据、按语言分写的复现步骤（Python `pytest -q <file> -k <name>`、Java `mvn -q -Dtest=<ClassName>#<testName> test`、C++ `./<gtest_exe> --gtest_filter=<Suite>.<Test>`） |
| EC-3 | 行为不一致或无法判断   | 差异报告、预期来源、待确认问题                                                                                                                                               |
| EC-4 | 连续自愈失败 3 轮   | 当前最佳版本、失败诊断、人工介入建议                                                                                                                                            |
| EC-5 | 达到最大轮次 20 轮  | 中间结果、已完成步骤、停止原因                                                                                                                                               |
| EC-6 | 工具执行不可恢复错误   | 错误说明、失败工具、建议处理方式                                                                                                                                              |

## 4 非功能需求

### 4.1 性能需求

| 编号          | 需求项     | 指标或约束                                                                                                    |

| NFR-PERF-04 | 大文件处理   | 超过500行或20个函数时给出规模警告，并支持分批摘要                                                                              |

| 编号         | 需求项    | 说明                                                                                         |
| ---------- | ------ | ------------------------------------------------------------------------------------------ |

| 编号           | 需求项     | 说明                                                                                                                                                |

### 5.2 数据保留与访问

会话期中间结果（解析中间产物、覆盖率原始报告、自愈诊断上下文等）通过 InMemoryStore 与 session-state 内存保存，进程结束即清理，不进入文件系统或数据库持久层。V1.0使用文件系统和本地JSON保存任务结果，适合个人本地使用。V3.0引入MySQL保存长期业务数据，引入Redis保存运行中消息、循环状态和工具调用中间结果。Redis中运行态默认保留24小时，MySQL业务数据默认永久保留，文件系统保留时间通过配置控制。

## 6 技术选型与实现约束

### 6.1 框架选型

本项目计划采用 **Mastra** 作为 Agent 应用开发框架。Mastra 负责 Agent 定义、工具注册、Workflow 编排、结构化工具调用、Guardrails（护栏）、人工审批、运行调试和后续 Web 服务集成；Python 继续负责 Python 生态强绑定的能力（pytest 执行、coverage.py 采集），Java/C++ 走 Language Adapter 通过 `command-runner` 调度外部 CLI（`mvn verify` / `g++` / `gcov` / `JaCoCo`）。

| 能力         | 技术选型                                  | 说明                                                       |
| ---------- | ------------------------------------- | -------------------------------------------------------- |
| Agent编排    | Mastra Agent（TypeScript）              | 负责逻辑摘要、测试策略、测试代码生成、失败诊断等 LLM 决策能力                        |
| 主流程控制      | Mastra Workflow（TypeScript）           | 固化解析、生成、执行、覆盖率、诊断、自愈、导出的主流程和退出条件                         |
| 工具定义       | Mastra Tool + Zod Schema（TypeScript）  | 为每个工具定义入参、出参、风险等级和执行函数                                   |
| Python 解析  | Python `ast`                          | Python 源代码解析                                             |
| Java 解析    | tree-sitter-java（TypeScript）          | Java 源代码解析（包名、类、方法、注解、modifier）                          |
| C++ 解析     | tree-sitter-cpp（TypeScript）           | C++ 源代码解析（类、方法、namespace、include）                        |
| Python 执行  | Python + pytest + subprocess          | pytest 原生执行环境，采集 stdout、stderr、exit\_code 和 traceback    |
| Java 执行    | mvn verify + JUnit 5 + JaCoCo（外部 CLI） | 一次调用产出测试结果与 JaCoCo 报告                                    |
| C++ 执行     | g++ + GoogleTest + gcov（外部 CLI）       | 编译 + gtest 二进制执行 + gcov 数据采集                             |
| Python 覆盖率 | coverage.py（Python Runtime）           | 通过 `coverage_runner.py` 调用                               |
| Java 覆盖率   | JaCoCo Maven Plugin 0.8.12            | `mvn verify` 同生命周期产出                                     |
| C++ 覆盖率    | gcov + gcovr                          | 由 cpp-adapter 解析 gcov 输出                                 |
| Web与API    | Mastra Server / Node.js Web框架         | V3.0阶段承载用户账户、任务创建、历史查询和结果下载                              |
| 持久化        | MySQL/Redis/InMemoryStore/文件系统        | 会话期中间结果用 InMemoryStore；Mastra Memory 辅助对话上下文；业务数据用自定义表结构 |

### 6.2 多语言架构约束

系统采用 **TypeScript 主 + 多语言外部命令** 架构。TypeScript 层承载 Mastra 项目、Agent、Workflow、工具注册、Web 接口、LLM 适配、Language Adapter 与 command-runner；Python 层承载 pytest 执行、coverage.py 采集；Java/C++ 通过 TypeScript 端 command-runner 调用外部 CLI（`mvn verify` / `g++` / `gcov` / `JaCoCo`）。

| 分层                | 实现方式                | 主要内容                                                                                   |
| ----------------- | ------------------- | -------------------------------------------------------------------------------------- |
| Agent与Workflow层   | TypeScript          | Mastra Agent、Workflow、Tool、Guardrails、Approval、Server                                  |
| Language Adapter层 | TypeScript          | python-adapter / java-adapter / cpp-adapter，统一 `ExecutionResult` / `SourceAnalysis` 接口 |
| 代码分析层             | Python / TypeScript | Python `ast`；Java/C++ 走 tree-sitter，由 TypeScript 调度                                    |
| 测试执行层             | Python / 外部 CLI     | pytest（Python）；`mvn verify`（Java）；`g++` + gtest 二进制（C++）                               |
| 覆盖率采集层            | Python / 外部 CLI     | coverage.py（Python）；JaCoCo（Java）；gcov/gcovr（C++）                                       |
| 数据与导出层            | TypeScript          | 按语言输出 `.py` / `.java` / `.cpp` 测试代码文件 + `.md` 文档，按目录分导出                                |
| Web展示层            | TypeScript          | V3.0阶段使用 React/Next.js 或 Mastra Server 集成                                              |

### 6.3 Mastra使用边界

Mastra用于解决 Agent 工程化和流程编排问题，但不承担生产级代码执行隔离。pytest / `mvn verify` / `g++` 执行均通过 `command-runner`（统一超时、路径白名单、风险评级、stdout/stderr 缓冲）调度，临时目录与路径白名单由 command-runner 强制。Mastra Memory 适合保存对话上下文和用户偏好，不替代 `agent_runs`、`test_code_versions`、`execution_results`、`diagnosis_results`、`coverage_results` 等业务数据表。CLI 端显式输出 API（`logAgent` / `logInfo` / `logWarn` / `logError` / `progress bar`）由 `cli-output` 模块封装，TTY/NO\_COLOR/CI 自动降级，不依赖全局 `console.log` 替换。

## 7 约束与假设

### 7.1 技术约束

1. LLM API延迟不可控，单次调用需10至60秒
2. LLM Tool Call能力依赖模型支持
3. Agent运行循环轮次越多，Token消耗越大
4. V2.1 起支持 Python 3.10+（pytest）、Java 17+（Maven 3.8+、JUnit 5）、C++ 17+（g++ 17+、GoogleTest 1.14+）三语；V1.0/V2.0 原边界"仅 Python"已被破除
5. V2.1 起执行隔离基于临时目录 + command-runner（subprocess 调度外部 CLI），不提供生产级恶意代码隔离
6. 多语言生成已纳入 V2.1 范围；多文件项目、跨模块依赖解析仍不在当前版本范围
7. 失败诊断依赖 traceback、surefire / JaCoCo / gtest 报告、测试代码版本、需求文本和 LLM 判断，无法保证 100% 准确，因此必须输出置信度和证据
8. Mastra 运行依赖 Node.js/TypeScript 生态；项目同时维护 Node.js、Python、Java（可选）、C++（可选）依赖

### 7.2 前置假设

1. 用户有LLM API的访问权限
2. V2.1 起各语言源代码均为可信代码；Java 适配器要求 Maven 能从 `pom.xml` 自动解析依赖；C++ 适配器要求 g++ 能找到头文件与库路径
3. 运行环境依赖：Node.js、Python 3.10+、pytest；V2.1+ 额外需要 Java 17+ / Maven 3.8+（Java 任务可选）、g++ 17+（C++ 任务可选）；V3.0 阶段额外需要 MySQL 8.0+ 和 Redis
4. 依赖推断：Python 走 `requirements.txt`、Java 走 `pom.xml`、C++ 走 `CMakeLists.txt` 或 `#include` 扫描；超出推断能力的依赖由用户在执行环境中提前安装
