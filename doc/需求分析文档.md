## 1 项目概述

### 1.1 项目背景

在软件开发生命周期中，测试用例设计和测试代码编写是保障软件质量的核心环节。然而，传统的人工方式面临四大痛点：

1. **效率低下**：开发工程师需逐条分析代码逻辑并手工编写测试用例和测试代码，耗时巨大
2. **质量参差**：不同工程师编写的测试用例粒度和测试代码覆盖率存在显著差异
3. **维护困难**：代码频繁变更时测试代码更新滞后
4. **调试低效**：测试失败时需要人工排查是源代码Bug还是测试代码自身的问题

为应对上述挑战，本系统研发一款基于大语言模型（LLM）的智能测试用例生成Agent。

### 1.2 项目定位

本项目实现的是一个具备自主决策能力的智能体（Agent），核心差异化特征：

1. **自主决策**：Agent根据输入内容和当前上下文，自主规划执行路径
2. **工具调用**：Agent通过调用预定义的工具集API完成具体操作，LLM负责决策"何时调用哪个工具"
3. **动态工作流**：每次执行路径可能不同，取决于源代码复杂度和测试结果
4. **循环运行与自愈**：Agent在循环中持续运行。测试失败时输出结构化诊断；测试代码错误则自动重生成并再次执行，源代码运行错误或行为不一致则向用户报告证据并等待确认

### 1.3 阶段规划

本项目范围收敛为三个可落地版本：

| 阶段   | 实现范围                              | 交付重点                 |
| ---- | --------------------------------- | -------------------- |
| V1.0 | Python 单文件、AST 解析、pytest 生成、执行、导出 | 先跑通从代码输入到测试文件输出的最短闭环 |
| V2.0 | 失败诊断、自愈循环、测试代码版本管理                | 处理执行失败，记录每轮生成版本和诊断证据 |
| V3.0 | Web 账户、历史记录、数据库持久化                | 支持多用户、历史任务查询、结果长期保存  |

各阶段之间存在依赖关系：V2.0 依赖 V1.0 的执行能力，V3.0 依赖 V1.0/V2.0 产生的结构化结果。

### 1.4 核心术语

| 术语                 | 定义                                                                 |
| ------------------ | ------------------------------------------------------------------ |
| Agent（智能体）         | LLM + 工具集 + 自主调用机制构成的自治系统                                          |
| Tool（工具）           | 可供Agent调用的函数API，每个工具有明确的输入输出规范                                     |
| Tool Call（工具调用）    | LLM在推理过程中决定调用某个工具并传入参数的行为                                          |
| Run Loop（运行循环）     | Agent持续执行的"感知→思考→行动→观察"循环                                          |
| Self-Healing（自愈循环） | 测试失败后自动判断根因、修复测试代码、重新执行的闭环                                         |
| Guardrail（护栏）      | 在高风险操作执行前进行的安全检查机制                                                 |
| Test Code（测试代码）    | Agent生成的、可直接被测试框架执行的单元测试代码                                         |
| AST（抽象语法树）         | Python 标准库 `ast` 解析得到的代码结构，用于稳定提取函数、类、参数和行号                        |
| Diagnosis（失败诊断）    | 对测试失败原因的结构化判断，包含类型、置信度、证据和建议动作                                     |
| Mastra             | TypeScript AI Agent框架，用于实现Agent、Tools、Workflows、Guardrails和可观测调试能力 |
| Workflow（工作流）      | Mastra中用于定义明确多步骤流程的编排机制，适合解析、生成、执行、诊断、自愈这类固定主链路                    |

### 2.3 代码解析

**需求描述**：Agent通过 `parse_source_code` 工具对上传的源代码文件进行结构化解析，提取以下信息：

- 模块名称和导入列表（import / from ... import）
- 类定义：类名、继承关系、类注释
- 方法/函数定义：函数名、参数列表（名称、类型注解、默认值）、返回类型注解、类注释、起始行号和结束行号
- 函数逻辑摘要（V1.0可选，V2.0起使用LLM生成）

**解析输出格式**（JSON结构）：

```json
{
  "module_name": "user_service",
  "imports": ["from flask import ..."],
  "classes": [
    {
      "name": "UserService",
      "bases": ["BaseService"],
      "docstring": "...",
      "methods": [
        {
          "name": "register_user",
          "params": [{"name": "phone", "type": "str", "default": null}],
          "return_type": "bool",
          "docstring": "...",
          "start_line": 15,
          "end_line": 42
        }
      ]
    }
  ],
  "functions": [...]
}
```

**边界情况**：

- 源文件存在语法错误时，抛出结构化错误，包含错误行号和原因，不进行后续生成
- 源文件中的 import 解析结果仅用于生成 mock/fixture 建议，不影响 AST 解析过程
- 空文件或仅含注释和 import 的文件：输出空结构，明确提示
- 大型文件（超过500行或20个函数）：输出文件规模警告，LLM按模块/类分批处理

### 2.4 测试生成

Agent根据AST解析结果和LLM逻辑摘要生成两部分内容：

**A. 测试用例文档**：用例编号、标题、优先级（P0/P1/P2/P3）、用例类型（功能/边界/异常）、前置条件、测试步骤、预期结果、关联方法。

**B. 测试代码**：可被测试框架直接执行的单元测试代码，例如：

```python
def test_register_user_success():
    result = register_user("13800001111", "test_user", "Pass1234", "123456")
    assert result == True

def test_register_user_duplicate():
    register_user("13800001111", "user1", "Pass1234", "123456")
    result = register_user("13800001111", "user2", "Pass5678", "654321")
    assert result == False
```

**生成策略**：功能测试（正常流程和分支路径）、边界测试（等价类划分和边界值分析）、异常测试（空值、超长输入、特殊字符、类型错误）。

### 2.5 测试执行与自愈

Agent调用 `execute_tests` 在临时执行目录中执行测试代码。执行结果包含：通过用例数、失败用例数、错误用例数、标准输出、标准错误、退出码、耗时、是否超时。

若测试失败，Agent调用 `diagnose_failure` 分析根因：

**情况A：`TEST_CODE_ERROR`**。Agent保存当前测试代码版本，重新生成下一版本测试代码、写入文件、再次执行。此过程最多自动重试3轮。

**情况B：`SOURCE_RUNTIME_ERROR`**。Agent输出错误报告：错误所在源文件和行号、异常类型和具体原因、pytest复现命令、traceback证据。

**情况C：`BEHAVIOR_MISMATCH`**。Agent输出实际结果与预期结果的差异，说明预期来自需求文本、类注释、函数名推断或LLM推断，不自动修改源代码。

**情况D：`UNKNOWN`**。Agent输出诊断证据和建议用户确认的问题，不进入自愈循环。

## 3 业务流程与运行机制

### 3.1 主业务流程

1. 用户通过CLI传入Python文件路径或粘贴代码；Web阶段也可通过页面上传文件或粘贴代码。
2. 系统读取源代码并进行路径、类型和基础大小检查。
3. Agent调用AST解析工具提取模块、类、函数、方法、参数、返回注解、类注释、行号和import信息。
4. Agent调用LLM生成逻辑摘要、输入约束、边界条件和测试关注点。
5. Agent生成测试用例文档，并据此生成pytest测试代码。
6. 系统保存测试代码版本，将代码写入临时执行目录和输出目录。
7. 系统运行pytest并采集执行结果。
8. 测试通过后执行质量检查；质量通过则导出测试用例、测试代码和执行报告。
9. 测试失败时进入失败诊断；仅当诊断为测试代码问题且置信度达到阈值时自动自愈。
10. 达到退出条件后返回最终结果、中间结果或需要用户确认的问题。

### 3.2 失败诊断业务规则

| 诊断类型                   | 典型证据                                                 | 系统处理                         |
| ---------------------- | ---------------------------------------------------- | ---------------------------- |
| `TEST_CODE_ERROR`      | traceback指向生成测试文件；fixture未定义；mock路径错误；调用不存在的函数或参数不匹配 | 当置信度不低于0.70时自动重生成测试代码        |
| `SOURCE_RUNTIME_ERROR` | traceback指向源代码文件；源代码语法错误、运行时异常或依赖缺失                  | 输出错误位置、异常原因、traceback证据和复现命令 |
| `BEHAVIOR_MISMATCH`    | 测试可执行，但实际结果与需求文本、docstring或用例预期不一致                   | 输出预期来源、实际结果和差异点，等待用户确认       |
| `UNKNOWN`              | 证据不足，无法可靠判断根因                                        | 输出已知证据和建议用户补充的信息，不自动修改       |

### 3.3 自愈边界

自愈仅处理测试代码自身问题，不修改用户源代码。系统最多连续自愈3轮，Agent总运行轮次最多20轮。若连续3轮仍失败，或诊断置信度不足，系统必须停止自动修改并报告当前版本、失败原因和人工介入建议。

### 3.4 退出条件

| 编号   | 退出条件        | 用户可见结果             |
| ---- | ----------- | ------------------ |
| EC-1 | 测试通过且质量检查通过 | 导出文件、测试代码、执行摘要     |
| EC-2 | 源代码运行错误     | 错误位置、原因、证据、复现步骤    |
| EC-3 | 行为不一致或无法判断  | 差异报告、预期来源、待确认问题    |
| EC-4 | 连续自愈失败3轮    | 当前最佳版本、失败诊断、人工介入建议 |
| EC-5 | 达到最大轮次20轮   | 中间结果、已完成步骤、停止原因    |
| EC-6 | 工具执行不可恢复错误  | 错误说明、失败工具、建议处理方式   |

## 4 非功能需求

### 4.1 性能需求

| 编号          | 需求项     | 指标或约束                       |
| ----------- | ------- | --------------------------- |
| NFR-PERF-01 | AST解析耗时 | 普通单文件（不超过500行）应在2秒内完成结构解析   |
| NFR-PERF-02 | 测试执行超时  | 默认60秒，用户可通过配置调整             |
| NFR-PERF-03 | LLM调用耗时 | 单次调用预期10至60秒，前端或CLI需显示等待状态  |
| NFR-PERF-04 | 大文件处理   | 超过500行或20个函数时给出规模警告，并支持分批摘要 |

### 4.2 可靠性需求

| 编号         | 需求项    | 说明                                         |
| ---------- | ------ | ------------------------------------------ |
| NFR-REL-01 | 结构化错误  | 语法错误、文件格式错误、LLM不可用、pytest超时均返回可读错误，不允许程序崩溃 |
| NFR-REL-02 | 结果可追溯  | 生成用例、测试代码版本、执行结果、诊断结果之间必须能建立关联             |
| NFR-REL-03 | 幂等导出   | 同一次任务重复导出不应破坏已有版本记录                        |
| NFR-REL-04 | 中间结果保留 | 失败退出时仍保留已生成的用例、测试代码和执行结果                   |

### 4.3 可维护性需求

| 编号           | 需求项     | 说明                                        |
| ------------ | ------- | ----------------------------------------- |
| NFR-MAINT-01 | 工具接口稳定  | Agent调用工具必须使用明确输入输出结构，便于替换LLM或执行环境        |
| NFR-MAINT-02 | 状态机可观测  | 每次状态流转记录step\_no、state、action和observation |
| NFR-MAINT-03 | 阶段可演进   | V1.0、V2.0、V3.0能力通过配置或模块边界逐步开启             |
| NFR-MAINT-04 | 测试质量可检查 | 最终测试代码不能只调用函数而没有有效断言，不能包含恒真断言作为主要验证       |

### 4.4 安全需求

| 编号         | 需求项     | 说明                             |
| ---------- | ------- | ------------------------------ |
| NFR-SEC-01 | 可信代码边界  | V1.0仅面向本地可信代码，不宣称生产级恶意代码隔离     |
| NFR-SEC-02 | 最小写入范围  | 工具只能写入临时目录或指定输出目录              |
| NFR-SEC-03 | 敏感配置保护  | Web阶段用户API Key必须加密存储，不在日志中明文输出 |
| NFR-SEC-04 | 高风险操作护栏 | 数据删除、覆盖外部路径等不可逆操作需要确认或拒绝       |

| 用户      | username、password\_hash、created\_at                              | Web阶段账户认证   |
| LLM配置   | api\_key\_encrypted、base\_url、model\_name                        | 每用户独立模型配置   |
| 源文件     | filename、language、content\_hash、file\_path                       | 记录输入代码来源    |
| 代码符号    | symbol\_type、name、signature、start\_line、end\_line、logic\_summary | 支撑用例生成和诊断定位 |
| Agent运行 | status、phase、self\_heal\_count、exit\_condition                   | 记录一次完整任务    |
| Agent步骤 | step\_no、state、action、observation                                | 记录状态流转      |
| 工具调用    | tool\_name、arguments、result、risk\_level、duration\_ms             | 审计和调试       |
| 测试用例    | case\_number、title、priority、case\_type、steps、expected\_result    | 生成文档和导出     |
| 测试代码版本  | version\_no、file\_path、content、status                            | 支撑自愈回溯      |
| 执行结果    | passed、failed、errors、stdout、stderr、exit\_code、timeout            | 判断任务结果      |
| 诊断结果    | diagnosis\_type、confidence、evidence、next\_action                 | 判断是否自愈      |
| 导出记录    | format、file\_path、created\_at                                    | 跟踪导出文件      |

### 5.2 数据保留与访问

V1.0使用文件系统和本地JSON保存任务结果，适合个人本地使用。V3.0引入MySQL保存长期业务数据，引入Redis保存运行中消息、循环状态和工具调用中间结果。Redis中运行态默认保留24小时，MySQL业务数据默认永久保留，文件系统保留时间通过配置控制。

## 6 技术选型与实现约束

### 6.1 框架选型

本项目计划采用 **Mastra** 作为Agent应用开发框架。Mastra负责Agent定义、工具注册、Workflow编排、结构化工具调用、Guardrails（护栏）、人工审批、运行调试和后续Web服务集成；Python继续负责源代码解析和pytest执行等与Python生态强绑定的能力。

| 能力           | 技术选型                                 | 说明                                                  |
| ------------ | ------------------------------------ | --------------------------------------------------- |
| Agent编排      | Mastra Agent（TypeScript）             | 负责逻辑摘要、测试策略、测试代码生成、失败诊断等LLM决策能力                     |
| 主流程控制        | Mastra Workflow（TypeScript）          | 固化解析、生成、执行、诊断、自愈、导出的主流程和退出条件                        |
| 工具定义         | Mastra Tool + Zod Schema（TypeScript） | 为每个工具定义入参、出参、风险等级和执行函数                              |
| Python AST解析 | Python `ast`                         | 保持对Python语法结构解析的准确性                                 |
| pytest执行     | Python + pytest + subprocess/容器      | 保留pytest原生执行环境，采集stdout、stderr、exit\_code和traceback |
| Web与API      | Mastra Server / Node.js Web框架        | V3.0阶段承载用户账户、任务创建、历史查询和结果下载                         |
| 持久化          | MySQL/Redis/文件系统                     | Mastra Memory可辅助对话上下文，业务数据仍以自定义表结构为准                |

### 6.2 双语言架构约束

系统采用 **TypeScript + Python** 双语言架构。TypeScript层承载Mastra项目、Agent、Workflow、工具注册、Web接口和LLM适配；Python层承载AST解析、pytest执行、测试运行沙箱和部分导出能力。两层之间通过标准输入输出、HTTP服务或本地子进程传递JSON数据。

| 分层              | 推荐语言              | 主要内容                                                  |
| --------------- | ----------------- | ----------------------------------------------------- |
| Agent与Workflow层 | TypeScript        | Mastra Agent、Workflow、Tool、Guardrails、Approval、Server |
| 代码分析层           | Python            | `parse_source_code`、语法错误定位、符号提取                       |
| 测试执行层           | Python            | pytest运行、临时目录、超时、traceback采集                          |
| 数据与导出层          | TypeScript/Python | 导出仅生成 `.py` 测试代码文件和 `.md` 测试用例/执行报告文档         |
| Web展示层          | TypeScript        | V3.0阶段使用React/Next.js或Mastra Server集成                 |

### 6.3 Mastra使用边界

Mastra用于解决Agent工程化和流程编排问题，但不承担生产级代码执行隔离。pytest执行仍需通过临时目录、超时、路径限制和后续容器沙箱独立实现。Mastra Memory适合保存对话上下文和用户偏好，不替代 `agent_runs`、`test_code_versions`、`execution_results`、`diagnosis_results` 等业务数据表。

## 7 约束与假设

### 7.1 技术约束

1. LLM API延迟不可控，单次调用需10至60秒
2. LLM Tool Call能力依赖模型支持
3. Agent运行循环轮次越多，Token消耗越大
4. V1.0仅支持Python 3.10+ 单文件源码和pytest框架
5. V1.0执行隔离基于临时目录和subprocess，不提供生产级恶意代码隔离
6. 多文件项目、跨模块依赖解析和多语言生成不纳入当前版本范围
7. 失败诊断依赖traceback、测试代码版本、需求文本和LLM判断，无法保证100%准确，因此必须输出置信度和证据
8. Mastra运行依赖Node.js/TypeScript生态，项目需要同时维护Node.js依赖和Python依赖

### 7.2 前置假设

1. 用户有LLM API的访问权限
2. V1.0阶段用户上传的源代码为可信Python代码，且不依赖复杂外部服务
3. 运行环境已安装Node.js、Python 3.10+ 和 pytest；V3.0阶段额外需要MySQL 8.0+ 和 Redis
4. 若代码依赖第三方库，用户需要在执行环境中提前安装依赖，或在后续版本补充依赖安装/环境管理能力
